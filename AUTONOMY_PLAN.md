# AUTONOMY_PLAN.md

## 目标

把当前项目的推进方式收敛成一套适合本地 `Windows 11` 项目的可持续工作流，让每一轮对话后都能稳定回答三件事：

1. 当前最重要的下一步是什么
2. 这一轮应该先做哪两个动作
3. 做完后如何继续衔接下一轮

---

## 核心原则

### 1. 先对齐文档，再推进

每一轮推进前，优先读取并对齐这些文件：

- `VISION.md`
- `FEATURES.md`
- `TODO.md`
- `docs/README.md`
- `docs/02-current-state.md`

### 2. 默认双任务推进

每一轮建议中：

- 给出 `3` 个最合适的下一阶段事项
- 默认执行前 `2` 个
- 做完后重新评估全局，再给出新的 `3` 个建议

### 3. 周期性 bug 环

连续推进若干轮后，进入 bug 环：

- 建议 1：查找 bug / flaky test / 状态漂移 / 性能异常
- 建议 2：修复 bug

### 4. 每轮必须留痕

每轮至少同步以下其一：

- `TODO.md`
- `docs/02-current-state.md`
- `docs/04-improvement-backlog.md`
- `docs/logs/YYYY/YYYY-MM.md`

### 5. 小步快跑，但必须可验证

默认小步推进；如果两个任务天然耦合，可合并为一轮中步推进，但仍需满足：

- 范围可控
- 有本地验证
- 可以明确说明结果与剩余风险

---

## 标准循环

### 阶段 A：读取与判断

每轮先做：

1. 读取 `VISION.md`
2. 读取 `FEATURES.md`
3. 读取 `TODO.md`
4. 读取 `docs/README.md` 与 `docs/02-current-state.md`
5. 判断当前最优先的 `3` 个下一阶段事项

### 阶段 B：执行前两个

默认执行：

- 建议 1
- 建议 2

执行要求：

- 优先修改最小必要范围
- 跑定向本地验证
- 必要时同步文档

### 阶段 C：再次发散

执行完后：

- 重新给出 `3` 个建议
- 再执行前两个

### 阶段 D：周期性 bug 环

当出现以下任一情况时，切到 bug 环：

- 已连续推进 `2~3` 轮功能
- 测试波动增多
- warning / flaky / 文档口径错位开始明显

修复完成后：

- commit
- 回到主循环

---

## 建议排序规则

建议列表按以下优先级排序：

1. 阻塞主线的事项
2. 对当前本地 Windows 产品价值最高的事项
3. 拖延成本最高的事项
4. 实现成本低但收益高的事项
5. 文档 / 测试 / 状态一致性事项

默认避免：

- 装饰性工作优先于主线工作
- 在旧问题未收口前大幅扩范围
- 文档长期滞后于代码

---

## 本项目适配说明

当前项目主循环重点：

1. Windows 本地产品化
2. Tauri 桌面壳搭建
3. Rust 核心与桌面服务层接缝
4. 本地验证与维护入口稳定化

因此，在没有更高优先级阻塞时，建议优先考虑：

- `desktop.ts` 统一调用层
- Tauri 命令面拆分
- 只读状态 / 任务 / 日志页面
- 本地验证和文档一致性
