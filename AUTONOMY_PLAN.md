# AUTONOMY_PLAN.md

## 目标

把当前项目推进方式收敛成一套可持续执行的工作流，让代理在每轮对话后都能：

1. 基于**最终目标文档 + 当前方向文档 + TODO/STATUS**判断全局最优先事项
2. 默认给出 **3–5 个下一阶段最合适做的事情**，按优先级排序
3. 默认执行前 **2 个**（除非用户打断/改规则）
4. 每完成一轮，再次发散并重新给出 **3–5 个建议**
5. 周期性进入 **bug 查找 / bug 修复** 阶段
6. 每轮保持文档、测试、提交同步，避免长期漂移

---

## 核心原则

### 1. 先读文档，再推进
每轮推进前，优先读取并对齐这些文件：

- `VISION.md`：最终目标，不许跑偏
- `CURRENT_DIRECTION.md`：当前阶段主线
- `TODO.md`：当前优先级清单
- `STATUS.md` / `PROGRESS.md`：当前真实落地状态
- 必要时补读：`PLAN.md`、`ROADMAP.md`、`FEATURES.md`、`GOLDEN_FEATURES.md`

### 2. 默认双任务推进
每一轮建议中：
- 给出 **3–5 个下一阶段最合适做的事情**
- 默认执行 **第 1 个 + 第 2 个**
- 做完后再重新评估全局，继续给出新的 3–5 个建议

### 3. 周期性 bug 环
每推进若干轮后，进入 bug 环：
- 建议 1：**查找 bug / flaky test / 状态漂移 / 性能异常**
- 建议 2：**修复 bug**
- 完成后 commit / push，再回到正常推进环

### 4. 每轮必须留痕
每轮至少同步以下其中之一：
- `TODO.md`
- `STATUS.md`
- `PROGRESS.md`
- `EXECUTION_LOG.md`
- 对应测试/提交记录

### 5. 小步快跑，但支持中步打包
默认小步推进；如果两个任务天然耦合，可合并成一轮中步推进，但仍必须：
- 范围可控
- 可验证
- 可回滚

---

## 标准循环（主循环）

### 阶段 A：读取与判断
每轮先做：
1. 读取 `VISION.md`
2. 读取 `CURRENT_DIRECTION.md`
3. 读取 `TODO.md`
4. 必要时读取 `STATUS.md` / `PROGRESS.md`
5. 判断当前最优先的 **3–5 个下一阶段事项**

### 阶段 B：执行前两个
默认执行：
- 建议 1
- 建议 2

执行要求：
- 优先修改最小必要范围
- 补测试
- 跑定向测试；必要时跑全量集成测试
- 若涉及结构变更，补文档口径

### 阶段 C：再次发散
执行完后：
- 从全局重新给出 **3–5 个建议**
- 再执行前两个

### 阶段 D：周期性 bug 环
当出现任一情况时，切到 bug 环：
- 已连续推进 2–3 轮功能
- 测试波动增多
- warning / flaky / drift / 文档口径错位开始明显

bug 环固定格式：
1. **查找 bug**
2. **修复 bug**

修复完成后：
- commit
- 若远程已配置且环境允许，则 push
- 再给出新的 3–5 个建议，回到主循环

---

## 建议排序规则

建议列表按以下优先级排序：
1. **阻塞主线的事项**
2. **主干价值最高的事项**
3. **返工风险最高、越拖越糟的事项**
4. **实现成本低但收益高的事项**
5. **文档/测试/状态一致性事项**

默认避免：
- 纯装饰性工作优先于主线工作
- 新开大功能前不收口旧能力
- 文档长期滞后于代码

---

## Bug 环细化规则

查 bug 时优先检查：
1. flaky tests
2. selection / verify / batch verify 结果漂移
3. 状态字段不一致（preview / explain / task / status）
4. SQL 写放大 / claim-reclaim 抖动
5. warning / panic / unwrap 风险点
6. 文档说有、代码没真正接上的能力

修 bug 时优先：
1. 先复现
2. 再最小修复
3. 再补测试锁住
4. 最后补文档/状态记录

---

## 重试与延迟策略

### 测试与异步状态查询
对可能异步收敛的场景（队列、批处理、后台状态）默认采用：
- **短轮询 + 小延迟**
- 例如 5–10 次轮询
- 每次间隔 50–250ms
- 先等稳定，再断言

### 外部/慢操作
对耗时操作默认：
- 避免立即判失败
- 优先定向测试，再全量测试
- 若失败先判断是否 flaky / 时序问题，再决定是否修主逻辑

### API/系统级操作
若涉及：
- 安装依赖
- 外部下载
- 系统配置
- 需要 root / 破坏性动作

则必须：
- 先判断必要性
- 能不装就不装
- 能局部替代就不全局安装
- 高风险时请求明确确认

---

## 安装/环境策略

只有在满足以下条件时才安装必要工具：
1. 当前任务明确依赖该工具
2. 无现成替代方案
3. 安装成本低于人工绕过成本
4. 不会破坏当前环境稳定性

安装后要记录：
- 为什么装
- 装了什么
- 是否影响 CI / 本机环境

---

## 文档同步策略

每 2–3 轮至少同步一次：
- `TODO.md`：真实优先级
- `STATUS.md`：当前阶段状态
- `PROGRESS.md`：只记录已落地能力

如果出现阶段变更：
- 先更新 `CURRENT_DIRECTION.md`
- 再更新 TODO / STATUS / PROGRESS

---

## 提交策略

默认：
- 每完成一轮稳定的小成果，就 commit
- commit message 必须说明这轮核心变化

在以下时机优先 commit：
1. 新能力刚闭环
2. flaky test 被稳定化
3. 文档口径已同步
4. bug 环修复完成

若远程已配置且环境允许：
- 在 bug 环或阶段性整理后可 push

---

## 输出策略

每轮对用户的输出默认包含：
1. **本轮结果**
2. **测试/验证结果**
3. **commit 信息（若有）**
4. **3–5 个下一阶段最合适做的事情**

排序要求：
- 第 1 项必须是当前最优先
- 默认紧凑，不长篇铺垫

## 当前项目适配说明（PersonaPilot）

当前项目主循环重点：
1. trust score 核心化
2. verify / smoke / batch verify 质量闭环
3. selection explainability 收口
4. 工程治理（warning、flaky、性能、文档一致性）

因此，在未来若无更高优先级阻塞，建议生成时优先考虑：
- explainability / diff 结构
- verify 质量信号主链接入
- flaky / warning / state drift 修复
- TODO / STATUS / PROGRESS 同步
