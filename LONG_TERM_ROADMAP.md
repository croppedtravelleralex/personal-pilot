# LONG_TERM_ROADMAP.md

`AutoOpenBrowser` 的中长期路线图。

目标：把项目从“可运行的浏览器自动化骨架”逐步演进为一个面向个人学习与研发、具备高并发、小头浏览器、高级指纹、强代理池、实验与观测能力的系统。

---

## 一、路线图分层

### A. 必须做（核心主线）
这些能力会决定项目能不能站住。

#### 1. 身份画像系统（Identity Profile）
从单一指纹配置升级为统一身份画像：
- 指纹
- 地区
- 语言
- 时区
- 设备类型
- 浏览器类型
- 网络策略

**为什么必须做：**
单点指纹参数不够，必须保证整体自洽。

#### 2. 指纹一致性校验器
在执行前检查：
- 指纹内部是否自洽
- 指纹与代理地区是否匹配
- 指纹与浏览器类型是否匹配
- 指纹与设备画像是否匹配

**为什么必须做：**
否则会产生“参数很多但看起来很假”的无效伪装。

#### 3. 代理质量评分系统
不只记录可用/不可用，而要记录：
- 成功率
- 延迟
- 失败趋势
- 目标站点适配度
- 风险等级

**为什么必须做：**
否则代理池无法真正优化，只会越堆越乱。

#### 4. 站点维度代理适配
同一代理对不同目标站点表现可能不同。
需要支持：
- 站点维度成功率
- 站点维度失败类型
- 地区 + 站点组合表现

**为什么必须做：**
后期调度的核心竞争力之一。

#### 5. 行为层模拟
逐步加入：
- 鼠标轨迹
- 点击节奏
- 输入节奏
- 滚动节奏
- 页面停留节奏

**为什么必须做：**
只有静态指纹，不足以支撑更真实的自动化身份。

#### 6. 会话连续性系统
支持维护：
- cookie
- localStorage
- sessionStorage
- referer 链路
- 浏览上下文

**为什么必须做：**
让访问更像“连续会话”而不是“瞬时脚本”。

#### 7. 策略引擎
把这些逻辑抽出来：
- 指纹选择
- 代理选择
- 地区匹配
- 并发阈值
- 补池规则
- 失败降级规则

**为什么必须做：**
避免 if/else 写死，影响后期扩展。

#### 8. 实验记录系统
记录每次实验：
- 使用了哪些 fingerprint / proxy / strategy
- 结果如何
- 成功率如何
- 延迟如何
- 哪些组合有效

**为什么必须做：**
这个项目是拿来学习与研发的，没有实验记录就无法真正积累结论。

---

### B. 应该做（增强主线）
这些能力会明显提升系统质量和研发效率。

#### 9. 代理生命周期管理
建议状态：
- candidate
- validated
- hot standby
- active
- degraded
- cooling
- evicted

#### 10. 可回放调试能力
支持保存：
- 任务上下文
- 截图
- HTML
- 日志
- 关键事件

#### 11. 指标面板 / 研究面板
展示：
- 当前并发
- 池中代理健康度
- 地区库存
- 任务成功率
- 验证成功率
- 指纹策略命中情况

#### 12. 多来源代理供给
支持：
- 开源代理源
- 手工导入
- 订阅源
- 后续供应商源

#### 13. 模块化 validator / harvester / runner
支持后续替换：
- 抓取器
- 验证器
- 浏览器执行引擎
- 指纹生成器

---

### C. 可以以后再做（远期能力）
这些不应阻塞早期闭环，但值得预留。

#### 14. 工作流引擎
把单任务执行升级为多步骤流程编排。

#### 15. webhook / callback 通知
任务完成后主动通知外部系统。

#### 16. 多用户 / 权限隔离
如果未来从个人研发平台演进为多人系统，需要补上。

#### 17. 更复杂的资源调度系统
例如：
- 浏览器实例池
- runner 级资源配额
- 高级并发控制

#### 18. 去中心化规则系统
让策略、画像、代理规则通过配置/策略文件驱动，而不是硬编码。

## 二、推荐推进顺序

### 第一阶段：建立最小闭环
- Rust 工程骨架
- SQLite schema
- REST API
- queue
- fake runner

### 第二阶段：把网络与身份抽象立起来
- fingerprint profile
- proxy model
- task network policy
- validator / allocation

### 第三阶段：强化系统智能性
- 身份画像系统
- 指纹一致性校验器
- 代理质量评分
- 地区策略
- 实验记录

### 第四阶段：增强真实感与研发能力
- 行为层模拟
- 会话连续性
- 调试回放
- 指标面板

### 第五阶段：平台化与长期演进
- 工作流引擎
- webhook
- 多用户/权限隔离
- 高级调度系统

## 四、路线图判断原则

后续如果要新增功能，优先看它是否满足以下任一条件：

1. 是否让系统更接近“真实身份 + 真实网络 + 真实行为”的统一画像
2. 是否让代理池更稳定、更聪明、更可持续
3. 是否让研发结论更可追踪、更可对比
4. 是否让架构更清晰、更易扩展
5. 是否显著提升验证与调试能力

如果都不满足，优先级就应该下调。
