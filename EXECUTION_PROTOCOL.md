# EXECUTION_PROTOCOL.md

`PersonaPilot` 的周期执行协议。

目标：把“每 5 分钟执行一次，持续 8 小时”的想法，落成一个可执行、可检查、可汇总、可防跑偏的项目推进协议。

---

## 1. 基本节奏

- **触发频率**：每 5 分钟一次
- **总时长**：8 小时
- **总轮次**：约 96 轮
- **执行单位**：4 轮组成一个 `mini-cycle`
- **周期时长**：20 分钟 / cycle
- **总周期数**：约 24 个 cycle

> 核心原则：不是每 5 分钟“随便做点什么”，而是按固定轮次类型推进。

---

## 2. 四轮制 mini-cycle

### Round 1 — 规划轮（Plan）
目标：收敛目标、更新上下文、只选一个主目标。

**必须做：**
- 读取：
  - `VISION.md`
  - `ROADMAP.md`
  - `STATUS.md`
  - `TODO.md`
  - `CURRENT_TASK.md`
  - `RUN_STATE.json`
- 更新 `ROADMAP.md`：过去 / 现在 / 未来
- 选定本轮唯一主目标
- 更新 `RUN_STATE.json`

**本轮不建议：**
- 开多个方向
- 做大改动
- 大量写代码

---

### Round 2 — 实现轮（Build）
目标：做一个最小但真实的推进动作。

**允许动作：**
- 写代码
- 改代码
- 新建模块
- 补 schema 草案
- 完善接口定义
- 补充设计文档
- 细化 proxy / fingerprint / runner 相关结构

**必须满足：**
- 只能有一个主目标
- 必须产生可见产出

**可接受产出：**
- 文件新增
- 文件修改
- 新增 schema 草案
- 新增模块骨架
- 新增测试草案

---

### Round 3 — 验证轮（Verify）
目标：做 bug 检查与功能验证，防止项目空转。

**优先动作：**
- `cargo check`（若工程已初始化）
- 模块结构一致性检查
- 文档与代码一致性检查
- schema 与模型一致性检查
- runner / queue / api 设计闭环检查
- 磁盘使用与落盘风险检查
- 记录当前 bug / 风险 / 缺口

**输出要求：**
- 把验证结果写入 `EXECUTION_LOG.md`
- 如有必要更新 `STATUS.md`

---

### Round 4 — 汇总轮（Summarize）
目标：总结前 4 轮推进结果，并向用户输出阶段汇总。

**必须做：**
- 汇总前 4 轮完成情况
- 更新 `ROADMAP.md`
- 更新 `EXECUTION_LOG.md`
- 输出飞书阶段汇总

**汇总格式建议：**
1. 本周期完成了什么
2. 当前卡点是什么
3. 风险/bug 是什么
4. 下一周期建议做什么

---

## 3. 8 小时分阶段推进

### 阶段 1（0h - 2h）
目标：设计收敛 + 文档对齐

优先事项：
- 补齐缺失设计文档
- 校准项目北极星
- 收敛当前主线
- 明确 schema 与模块边界草案

### 阶段 2（2h - 4h）
目标：工程骨架落地

优先事项：
- 初始化 Rust 工程
- 建立 `src/` 模块结构
- 建立 `Cargo.toml`
- 建立 `main.rs / lib.rs`
- 落最小目录骨架

### 阶段 3（4h - 6h）
目标：最小闭环搭建

优先事项：
- 任务模型
- queue 骨架
- runner trait
- fake runner
- API 最小路由骨架

### 阶段 4（6h - 8h）
目标：验证与收敛

优先事项：
- bug 检查
- 功能验证
- 文档回填
- 阶段总结
- 下阶段工作计划

---

## 4. 强约束（防跑偏规则）

### 规则 1：每轮最多一个主目标
防止一轮内同时推进多个方向。

### 规则 2：每轮必须有产出
无论是文档、代码、schema、验证结果，都必须落到文件中。

### 规则 3：偶数轮优先收敛，不优先扩需求
尤其验证轮，重点是收敛与检查，而不是继续发散。

### 规则 4：禁止连续 3 轮只写文档不触达工程
项目已经有较多文档，后续必须逐渐进入工程骨架阶段。

### 规则 5：每 4 轮必须回答一次
> 项目距离最终效果更近了吗？

如果答不上来，说明本周期可能在空转。

---

## 5. 每轮执行前必须读取的文件

- `VISION.md`
- `ROADMAP.md`
- `STATUS.md`
- `TODO.md`
- `CURRENT_TASK.md`
- `CURRENT_DIRECTION.md`
- `RUN_STATE.json`

按需要读取：
- `DESIGN_NETWORK_IDENTITY.md`
- `LONG_TERM_ROADMAP.md`
- `GOLDEN_FEATURES.md`
- 其他专题设计文档

---

## 6. 当前项目的推荐执行焦点

在当前阶段，周期执行时优先级建议如下：

### P0
- 开源项目参考清单
- schema 草案
- Rust 工程骨架
- 模块目录结构

### P1
- task model
- queue model
- runner trait
- network_identity 模块占位

### P2
- fake runner 骨架
- 最小 API 骨架
- 初步验证流程

---

## 7. 飞书阶段汇总规范

每 4 轮后向用户发送一次简洁汇总，格式：

- **本周期完成**：列 2-4 条
- **当前问题**：列 1-3 条
- **当前状态**：一句话总结
- **下一步**：列 1-2 条

**避免：**
- 过长长文
- 过度重复
- 只有空泛鼓励没有实质内容

---

## 8. 终止条件

满足以下任一条件时应停止本轮自动推进：

- 8 小时周期结束
- 用户显式要求停止
- 连续多轮验证失败且没有新产出
- 项目进入需要人工决策的关键分叉点

---

## 9. 当前建议

当前不建议立即恢复“每 5 分钟自动推进”，而是：

1. 先把本协议落地
2. 先补足工程骨架
3. 再根据该协议恢复自动执行

这样后续自动推进的质量会高很多。
